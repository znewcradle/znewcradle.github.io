垃圾收集与内存分配策略

发表于 2019-04-21 | 分类于 java |

垃圾收集器与内存分配策略

1.概述

  • 线程隔离的的数据区域：虚拟机栈、程序计数器和本地方法栈都是随着线程而生灭，虚拟机栈中栈帧在类结构确定的时候已知，有条不紊地出栈和入栈——内存确定
  • 方法区内存同样是确定的；只有堆内存是在运行期间会进行动态分配和回收，成为GC的主战场

  1. 对象消亡判断

    • 引用计数算法：难以解决对象之间相互循环引用问题，未被JVM采用
    • 可达性分析算法：通过一系列GC Roots对象作为起点，向下搜索，经过引用链如果不可达则对象不可用会被回收，其中GC Roots对象包括：
      • 虚拟机栈（栈帧中的本地变量表）引用的对象
      • 方法区中类静态属性引用的对象
      • 方法区中常量引用的对象
      • 本地方法栈中JNI（Native方法）引用的对象
    • JDK1.2之前只有：被引用和未被引用两种状态；之后，引入了4种引用强度一次减弱的引用类型，用于描述可留可不留的对象，如缓存
      • 强引用：代码中普遍存在，GC不会回收
      • 软引用：有用但并非必须的对象
      • 弱引用：非必须对香港
      • 虚引用：虚引用的存在完全不会对其生存时间构成影响
    • 对象至少经历两次标记过程
      • 对象不可达后被第一次标记并筛选，判断此对象是否有必要执行finalize方法；如果对象没有覆盖finalize或者finalize已经被JVM调用过，则没必要；
      • 如果被徐选中，则放入F-Queue队列中，帮在稍后由JVM创建的、低优先级的Finalizer线程去执行
    • 回收方法区：
      • 回收废弃常量：当前系统中任何对象引用
      • 回收无用的类：要满足所有的实例都已经被回收，加载该类的ClassLoader已经被回收，该类对应的java.lang.Class对象没有在任何地方被引用，无法在任何地方通过反射访问该类的方法。虚拟机提供了-Xnoclassgc参数进行控制，尤其是频繁自定义ClassLoader的场景需要具备类卸载功能

  2. 垃圾收集算法

    • 标记-清除算法：先标记所有需要回收的对象，再统一回收；存在的缺陷
      • 效率问题：标记和清除的效率不高
      • 空间问题：标记清除后会产生大量不连续的内存碎片
    • 复制算法：将内存分为8:1:1的一块大Eden空间和两块Survior空间；回收时，将Eden和Survivor中存活着的对象一次性地复制到另外一块Survivor空间上，最后清理掉Eden+老Survivor。同时，由老年代内存进行分配担保（其他内存）——基于新生代对象98%是朝生夕死
    • 标记-整理算法：标记后，将所有存活对象都移动到一端，然后清理边界以外的内存（新生代对象存活率较高）
    • 分代收集算法：根据对象存活周期划分内存，根据不同年代采用特定收集算法；
      • 新生代用复制算法（存活率低）
      • 老年代用标记-清理或者标记-整理（存活率高）

  3. HotSpot算法实现

    • 枚举根节点：可达性分析中，方法区庞大 +ＧＣ停顿（保证分析结果一致性和准确性）；在HotSpot中使用OopMap的数据结构来在类加载过程中，计算对象内数据类型和偏移量关系，方便GC在扫描时得知
    • 安全点：程序执行时并非在所有地方都停顿下来开始GC，只在安全点才如此；选定安全点的原则是：是否让程序长时间执行的特征为标准，特征是指令序列复用，如：方法调用；GC时让所有线程停顿的方案：抢先式中断（没有JVM采用） + 主动式中断（设置标志，让线程轮询，主动中断挂起）
    • 安全区域：被扩展的安全点，指一段代码片段中，引用关系不会发生变化；在离开安全区域时，需等待GC完成

  4. 垃圾收集器

    • 七种作用于不同分代的收集器：Serial, CMS, Serial Old, ParNew, Parallel Scavenge, Parallel Old, G1

    • Serial收集器：最基本和悠久的，在垃圾收集时，必须暂停其它工作线程，直到收集结束；简单高效尤其是单CPU环境；对新生代采取复制，对老年代采用标记-整理

    • ParNew收集器：

      • Serial收集器的多线程版本，除了多线程特点之外其余相似；

      • 但是只有它能与CMS收集器配合工作（并发收集器——不用暂停工作线程）；在JDK中使用CMS收集老年代时，新生代只能选择ParNew或者Serial

      • 在单CPU下因为线程交互，反而效率下降；随着CPU数量增多才会体现高效

      • 并行：多条垃圾收集线程并行工作，此时用户线程处于等待状态；并发：指用户线程与垃圾收集线程同时执行（不一定并行可能是交替执行），用户线程继续运行，而垃圾收集程序运行于另一个CPU上。

    • Parallel Scavenge收集器（吞吐量=用户代码运行时间/（用户代码运行时间+垃圾收集时间））

      • 新生代收集器，使用复制算法，并行多线程，与ParNew的区别在于达到可控制的吞吐量，高吞吐量有利于用户高效利用CPU——吞吐量优先收集器
      • 控制最大垃圾收集停顿时间：-XX: MaxGCPauseMillis（但是过小会导致收集次数会增加）以及直接设置吞吐量大小的-XX: GCTimeRatio参数（0-100之间，垃圾收集时间占总时间的比率）
      • 提供-XX: +UseAdaptivePolicy使用GC自适应调节策略来根据当前系统的运行情况收集性能监控信息，动态调整参数

    • Serial Old收集器

      • Serial收集器的老年代版本，单线程，使用“标记——整理”算法，主要给Client模式下的JVM使用；Server模式下主要用于：1. 在JDK1.5及以前版本中与Parallel Scavenge收集器搭配使用； 2. 作为CMS收集器的后备预案（并发收集发生Concurrent Mode Failure时）
      • Paralel Scavenge的收集器架构中本身有PS MarkSweep收集器进行老年代收集，但与Serial Old非常接近

    • Parallel Old

      • Parallel Old是Parallel Scavenge收集器的老年代版本，多线程和“标记——整理”算法
      • JDk1.6中出现，取代Serial Old配合Parallel Scavenge

    • CMS收集器——Concurrent Low Pause Collector并发低停顿收集器）

      • 目标是获取最短回收停顿时间，基于“标记——清除“算法，四个阶段：初始标记-> 并发标记->重新标记->并发清除
      • 初始标记、重新标记，需要”Stop The World”，初始标记用来标记GC Roots能直接关联到的对象；重新标记：为了修正并发标记期间因用户程序继续运作而导致标记产生变动的那部分对象的标记
      • 并发标记：进行GC RootsTracing过程；并发标记和并发清除用时最长，但是可以与用户线程一起工作
      • CMS对CPU资源敏感，占用部分线程，总吞吐量会降低
      • CMS无法处理浮动垃圾，出现Concurrent Mode Failure，即并发清理阶段，用户线程继续运行而产生的垃圾，只能下一次GC时清理，称为浮动垃圾；如果老年代占用了一定空间后CMS会被激活可以通过-XX: CMSInitiatingOccupancyFraction。如果CMS运行期间预留的内存不足，会出现Concurrent Mode Failure，JVM启动Serial Old，导致另一次Full GC
      • 标记——清除会产生空间碎片，用-XX： +UseCMSCompactAtFullCollection开启FullGC过程进行内存碎片合并；用-XX: CMSFullGCsBeforeCompaction用于设置多少次FullGC后，进行一次带内存压缩的FullGC

    • G1收集器

      • 面向服务端，并行与并发（多CPU缩短Stop-The-World的时间），分代收集，空间整合（基于”标记-整理“算法），可预测的停顿（可指定垃圾回收用时）
      • G1的新生代和老年代在Region中物理内存连续但做标记；维护优先队列跟踪各个Region垃圾堆的价值大小；
      • G1收集器运作：初始标记（Stop-The-World） -> 并发标记 -> 最终标记 -> 筛选回收；最终标记是修正并发标记期间因用户程序继续运作而导致标记产生变动的部分标记记录；筛选回收对各个Region的回收价值和成本进行排序，根据用户期望GC停顿时间

    • GC日志

    • 垃圾收集器参数总结

  5. 内存分配与回收策略

    • 对象优先在Eden分配
      • 对象在新生代Eden区分配，如果空间不足则JVM发起MinorGC
      • 新生代GC（Minor GC）频繁且速度快；老年代GC（Major GC/ Full GC）经常伴随至少一次MinorGC，速度比Minor GC慢10倍以上
    • 大对象直接进入老年代
      • 需要大量连续内存空间的Java对象，如：长字符串和数组
      • -XX: PretenureSizeThreshold参数令大于该值的对象直接在老年代分配

    • 长期存活的对象将进入老年代

      • 对象从Eden出生到第一次Minor GC后存活并被Survivor容纳，对象年龄设为1；之后，每熬过一次Minor GC，年龄增加1；通过-XX: MaxTenuringThreshold设置老年代的年龄阈值，默认15

    • 动态对象年龄判定

      • 忽略MaxTenuringThreshold，如果Survivor空间中相同年龄所有对象大小的总和大于Survivor空间的一半，年龄大于或等于该年龄的对象可直接进入老年代
    • 空间分配担保
      • JVM检查老年代最大可用的连续空间是否大于新生代所有对象总空间，如果条件成立，则Minor GC确保安全；否则，JVM查看HandlePromotionFailure设置是否允许担保失败
      • 如果允许担保失败，则检查老年代最大可用的连续空间是否大于历次晋升到老年代对象的平均大小，成功则有风险的MinorGC；反之，进行Full GC
      • 如果担保失败，或者HandlePromotionFailure都会重新发起Full GC

算法设计思想

贪婪算法

  1. 解释：分阶段工作，并且在每个阶段寻找当前最优，即局部最优；当算法终止时，局部最优等于全局最优。有时，不需要最佳答案，也可以用贪婪算法生成近似解

  2. 典型问题

    • 简单调度问题

      使用最短作业优先算法，每个阶段都是寻找当前阶段用时最少的作业，从局部最优到最后形成了全局最优

    • 哈夫曼编码

      • 哈夫曼编码树是满树（所有节点要么是树叶，要么有两个儿子）；字符放在树叶上，从根节点到树叶的路径字符编码，形成前缀码；从根节点往左走得到’0’，往右走得到’1’
      • 算法对由树组成的一个森林进行。一棵树的权等于它的树叶的频率的和。任意选取最小权的两棵树T1和T2，并任意形成以T1和T2为子树的新树，将这样的过程进行C-1次。在算法的开始，存在C棵单节点树（每个字符一棵），算法结束时形成最优哈夫曼编码树
      • 从算法描述上来看，哈夫曼编码仍是一个典型的贪婪算法问题，因为每次都是选取当前权重最小的两棵树进行合并，形成新树；最后形成的树就是全局最优的编码树
      • 不过，该算法需要两趟扫描；其次，压缩问题件的开头需传送编码信息，对于小文件而言，传送编码的信息代价将超过压缩中任何可能的节省，可能导致文件扩大。

    • 近似装箱问题

      • 联机装箱算法：下项适合算法、首次适合算法、最佳适合算法
      • 脱机装箱算法：首次适合递减算法、最佳适合递减算法

分支算法

  1. 分支算法分为两个部分
    • 分：递归解决较小的问题（基本情况除外）
    • 治：从子问题的解构建原问题的解
  2. 至少含有两个递归调用的例程叫做分支算法，一般子问题不相交，典型问题是：归并排序和快速排序
  3. 典型问题
    • 最近点问题：将平面上的点用直线划分为三个部分，分别是直线的左右两个部分和跨直线的点，可以得到O(N logN)算法
    • 选择问题：找出N个元素的集合中第k个最小的元素
    • 算术问题的理论改进
      • 整数相乘
      • 矩阵乘法

动态规划

  1. 任何数学地推公式可以直接转换为递归算法，但是编译器对于递归算法不能正确对待（编译低效）导致低效的程序。给编译器提供帮助，将递归算法改写成非递归算法，将子问题的答案系统地记录在表内，形成了一种技巧——动态规划

    核心问题：用表代替递归

  2. 典型问题：

    • 斐波那契数列计算：如果使用递归的方式，就会做重复工作，通过保留中间结果可以避免重复结果的计算，提高运行速率（空间换时间）
    • 矩阵乘法的顺序安排：
    • 最优二叉查找树：
    • 所有点对的最短路径

随机化算法

  1. 在算法中使用随机函数，且随机函数的返回值直接或者间接的影响了算法的执行流程或执行结果；即算法在运行的过程中某一步或某几步涉及一个随机决策（一个决策依赖于某种随机时间），很典型的如：快速排序

回溯算法

  1. 相当于穷举搜索的巧妙发现，但性能一般不理想，不过相对于蛮力穷举搜索的工作量有显著的节省
  2. 回溯问题的程序存在一个典型模板，相对来说比较稳定；回溯方法的签名中的传入参数一般有：最终结果，暂存结果，输入，（问题相关的变量值）
  3. 为了优化回溯问题的性能，要适当的剪枝

动态规划总结

参考链接：

  • https://pks-loving.blog.luogu.org/junior-dynamic-programming-dong-tai-gui-hua-chu-bu-ge-zhong-zi-xu-lie
  • https://www.zhihu.com/question/23995189?utm_source=com.microsoft.launcher&utm_medium=social&utm_oi=719823168826523648

动态规划求解的一般思路：

  1. 判断问题的子结构，当具有最优子结构时，动态规划可能适用
  2. 求解重叠子问题，一个递归算法不断地调用同一问题，递归可以转化为查表从而利用子问题的解。
  3. 重新构造一个最优解
区别
  1. 与贪心策略的区别：贪心策略值考虑眼前的情况；动态规划会考虑下一步的值；
  2. 与暴力的区别：暴力求解的解空间往往非常大，基本上所有可能解；动态规划则自带剪枝，避免求解冗余信息

每个阶段只有一个状态 -> 递推；

每个阶段的最优状态都是又上一阶段的最优状态得到的 -> 贪心

每个阶段的最优状态是由之前所有阶段的状态的组合得到的 -> 搜索

每个阶段的最优状态可以从之前某个或者某些状态直接得到而不管之前这个状态是如何得到的 -> 动态规划

动态规划：将一个问题拆成几个子问题，分别求解这些子问题，即可推断出大问题的解

一言以蔽之：大事化小，小事化了

重要的性质：
  1. 无后效性：在给定某一阶段的状态后，则在这一阶段以后过程的发展不受这阶段以前各阶段状态的影响；
  2. 最优子结构：大问题的最优解可以由小问题的最优解推出；
如何判断一个问题是否能用DP来解决：
  1. 能够将大问题拆成几个小问题；
  2. 满足无后效性：求解f(n)只需要求解f(n-1)和f(n-2)，但是f(n-1)和f(n-2)的求解过程不影响对求解f(n)不造成影响，只需要知道其值即可 —— 当前状态会且仅会决策出下一状态，而不直接对未来的所有状态负责（Future never has to do with past time, but present does.）
  3. 最优子结构性质；（可以通过反证法归谬来证明，如果有一步不是最优子结构，那么就能找到比当前结果更优的解，与题设矛盾）—— 类似递推，但是针对状态操作
动态规划算法设计

通用步骤：

  1. 将当前面对的局面记为x —— 设计状态（我是谁）
  2. 对于状态x，记当前局面求解为f(x)，目标是求出f(T)
  3. 找出f(x)与哪些局面有关，记为p，可以得到状态转移方程：通过f(p)来推出f(x)

关键在于状态转移方程，可以通过“我从哪里来”（pull型转移）或“我要到哪里去”（push型转移）来解决

二分查找问题总结

Search Insert Position

之所以会返回lo，因为如果跳出循环，必然存在lo > hi，而target大于此时lo的前一个值，则必然是比前一个lo更大的值，结合二分查找，即现在的lo。

Find First and Last Position of Element in Sorted Array

这个题目还是很清晰的；利用二分查找，如果target不在数组中，则first和last的值不会改变仍旧是-1，最后返回即可。如果，在数组中找到该值，则首先为first和last都赋值mid；然后将first往前移，last往后移动，直到它们指向的值不等于target，此时得到结果，可以break。

Search in Rotated Array

这是二分查找的变形题目，从题意来看，是将原本有序的数组从某一处截断然后旋转，最终数组是两个部分分别有序，且后一部分都小于前一部分。在这种情形下，仍旧可以应用二分查找。

  1. 首先判断特殊情况，数组中的元素只有一个，则只会存在返回0和-1的情况
  2. 对于一般情形，先还是按照二分查找的一般套路来；其中复杂的判断式

int num = (nums[mid] < nums[0]) == (target < nums[0]) ? nums[mid] ​ : target < nums[0] ? Integer.MIN_VALUE : Integer.MAX_VALUE;

  • 如果nums[mid] < nums[0]且target < nums[0]，即此时，mid已经进入了后半部分中且target显然也位于后半部分，此时让num = nums[mid]；最后的lo, hi变换中，正常进行判断，跳过不必要的检查
  • 如果nums[mid] > nums[0]且target > nums[0]，即此时，mid进入了前半部分，且target位于前半部分，此时让num = nums[mid]；最后的lo, hi变换中，正常进行判断，跳过不必要的检查
  • 如果nums[mid] < nums[0]且target > nums[0]，即此时，mid已经进入了后半部分且target位于前半部分，让num=Integer.MAX_VALUE；最后的lo, hi变换中必然有，hi = mid, 跳过了mid之后的数组元素检查
  • 如果nums[mid] > nums[0] 且 target < nums[0]，即此时，mid已经进入了前半部分且target位于后半部分，让num=Integer.MIN_VALUE；最后的lo, hi变换中必然有，lo = mid + 1, 跳过了mid之前的数组元素检查

（以上涵盖了四种不同的情况）

总结

从最后一个变形的二分查找题目可以知道，二分查找本质上是让我们利用“有序”的特点来跳过某些不必要的检查；所以要仔细思考清楚如何利用“有序”的特点和检查结果来跳过不必要检查。（可以思考分情况讨论）

回溯问题总结

发表于 2018-11-25 | 分类于 回溯 ， algorithm ， algo-summary |

回溯问题总结

Combination Sum

由于数组中不存在重复，且可以重复使用数组中的元素，所以只要注意将该步迭代加入的元素的下标作为标记直接传递给下次迭代即可。这样就可以保证所有的迭代过程中产生的解必然是独一无二的。

Combination Sum II

相对于上一个题目，多了一个要求就是所有元素只能被使用一次，此时增加了三个操作：

  • 对数组进行排序，方便之后“去重”
  • 通过判断是否是重复元素，跳过不必要的添加
  • 将迭代参数的i换成了i + 1保证了所有元素只会被使用一次

如果使用标记数组仍旧会存在重复使用的情形，只有通过数组排序后，检查当前元素是否和上一次元素一致来跳过迭代。但同时，重复元素也还是保证了能够被同时用到，因为经过了排序，所以如果是连续取用重复元素是可以的，同时也可以保证取完；但是如果是跳跃着选元素，就会被阻止（这样也就是重复情形）

Permutations

数组中的元素是独一无二的，但是不能通过位置标记start来避免重复，因为要产生所有的排列，必然还是需要每次从头开始遍历。

如果判断是否contains的代价很高，可以利用空间换时间，用boolean[] marked来标记被访问过的元素，可以迅速得到被访问与否的判断。

  1. 位置标记start: 适合用来处理在每次迭代中遍历数组“不回头”的情形
  2. 数组标记boolean[] marked: 适合用来处理每次迭代过程中仍旧会“回头”遍历的情形

Permutations II

问题改动：数组元素存在重复，但是最后的排列不能重复；相对于上一题的改动

  1. 数组进行排序，方便之后”去重“

  2. 增加了访问数组，首先用是否访问来避免添加已经被添加过的元素，利用i > 0 && nums[i] == nums[i - 1] && !marked[i - 1]来避免重复添加和已经被添加元素相等的元素

    i > 0 && nums[i] == nums[i - 1] && !marked[i - 1]

    这个核心判断，保证了如果是直接的从一开始就添加重复元素，是允许的，如果是跳过一个再添加则被阻止，如果可以跳过一个再添加证明本身是重复添加，所以被阻止。

Subsets

数组中元素是唯一的，返回所有可能的子集合；只需要位置标记start保证不回头即可

Subsets II

题目改动：数组元素存在重复，但是子集不能存在重复，对应的策略

  1. 数组排序； 2. 仍旧是阻止”跳着“添加重复元素，只允许连续添加重复元素

Palindrome Partitioning

这里使用了位置标记start，显然是”不回头“的搜索，这样可以避免重复添加；同时这里的添加检查只是更为复杂，之前的添加检查都是单纯的重复检查，这里结合题目要求增加了是否是Palindrome Partitioning的检查；本质上和以上的题目是一致的。

Sudoku Solver

这道题目仍旧是回溯法题目，只是回溯法的问题空间从一维升高到了二维之中。首先是棋盘本身是二维的空间，在添加一个元素时，需要考虑添加1-9中的任一元素；这里通过checkValid判断添加之后棋盘是否仍旧满足数独规则来增添元素，而重复性检查通被board[i][j] == '.'所替代。

N-Queens

和数独问题十分类似，只是问题域是二维空间，多了一层嵌套循环而已，同时既有row标记的位置，又有业务检查

N-Queens II

同样是利用了回溯解法，只是利用了只用求解解的个数的特点，减少了一层循环。

总结

回溯问题无论问题域是一维空间或是二维空间都具有其规律性，和普遍的框架，即：

  1. 判断是否符合条件，加入到解集合中；
  2. 遍历问题域，构造解并进行重复性检查和业务检查
  3. 设置域，将构造解带入递归调用
  4. 回退，重置域，进入下一步迭代

在回溯过程中的重复问题，常用的解决手段有：

  1. 位置标记start：用于问题域遍历”不回头“的回溯问题
  2. 访问数组boolean[]：用于问题域遍历”回头”的回溯问题
